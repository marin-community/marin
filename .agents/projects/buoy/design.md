# buoy — a better viewer for wandb runs

_Why are we doing this? What's the benefit?_

We want a **better viewer** over wandb runs. wandb stays the source of truth; **buoy**, an Iris-hosted web app, lets anyone pick a run, pull **all** of its data — metrics, config, code, artifacts, and TPU profiles — into a GCS cache on demand, and then browse metric plots and the real xprof profile UI in one place. The GCS copy is a refetchable cache (TTL-cleaned), not an archive: anything missing or stale is re-pulled from wandb. The immediate driver is profiling — today xprof artifacts live in wandb and can only be inspected by hand-downloading `.xplane.pb` and running a local profiler. This service makes "look at this run's profile" a link instead of a 10-minute ritual.

## Background

Prior art and patterns are captured in [`research.md`](./research.md). Key facts: the repo's standard web-service shape is **Starlette + a Vue SPA**; **[finelog](https://github.com/marin-community/marin/blob/05a1d4d474b9dccc8189c031c24cc6452c2e589a/lib/finelog/OPS.md#L3) is the model for an independent service registered with Iris** (the controller dashboard at [`dashboard.py`](https://github.com/marin-community/marin/blob/05a1d4d474b9dccc8189c031c24cc6452c2e589a/lib/iris/src/iris/cluster/controller/dashboard.py#L384) is built *into* the controller and is the exception, not the rule). Finelog registers via the cluster-config `endpoints` map ([`config.py:582`](https://github.com/marin-community/marin/blob/05a1d4d474b9dccc8189c031c24cc6452c2e589a/lib/iris/src/iris/cluster/config.py#L582)) resolved at controller startup and is reached through the endpoint proxy at `/proxy/system.<name>/` ([`endpoint_proxy.py`](https://github.com/marin-community/marin/blob/05a1d4d474b9dccc8189c031c24cc6452c2e589a/lib/iris/src/iris/cluster/controller/endpoint_proxy.py#L1)); the lighter alternative is a job that self-registers at runtime via `ctx.registry.register(...)` ([`quick_serve.py:209`](https://github.com/marin-community/marin/blob/05a1d4d474b9dccc8189c031c24cc6452c2e589a/lib/marin/src/marin/inference/quick_serve.py#L209)). wandb is already read via `wandb.Api()`, with a profile-download example in [`marin/profiling/ingest.py`](https://github.com/marin-community/marin/blob/05a1d4d474b9dccc8189c031c24cc6452c2e589a/lib/marin/src/marin/profiling/ingest.py#L54). GCS paths use [`marin_temp_bucket`](https://github.com/marin-community/marin/blob/05a1d4d474b9dccc8189c031c24cc6452c2e589a/lib/rigging/src/rigging/filesystem.py). Plotly is the house charting lib.

## Challenges

- **xprof behind a two-hop proxy — spiked, and it works.** The profile UI is standalone `xprof` (OpenXLA, v2.22.3), a web server reading a logdir. It has **no `--path_prefix` flag**, but it doesn't need one: its `index.html` uses **relative** asset URLs plus a JS-computed `<base href>` derived from `document.location.pathname` (it splits on the literal `/data/plugin/profile`). So a **dumb forwarding reverse proxy** (no body rewriting) serving xprof under `/proxy/system.buoy/xprof/{run_key}/…` self-prefixes correctly across both proxy hops — verified: every asset/data URL the browser computes returns 200 through the prefix. The one requirement: the iframe `src` must point at `…/xprof/{run_key}/data/plugin/profile/` (not the bare app root) so the base-href logic triggers. Caveat: xprof's trace viewer frames `https://ui.perfetto.dev` and loads charts from `gstatic.com`, so the **viewer's browser needs public internet** — xprof is not fully offline.
- **xprof mutates the parent frame — confirmed on the prod deploy.** xprof is a TensorBoard/Colab *plugin*: its bundle writes its run/tag/host selection into `window.parent.history` (`pushState`/`replaceState`) and reads `window.parent.location` for deep-linking (it even sniffs `window.parent.TENSORBOARD_ENV`). Iframed directly into the buoy shell, opening a profile **clobbers the shell's own `?entity&project&run` query** with xprof's `?run&tag&hosts`, so a reload loses the buoy deep-link. It targets the *immediate* parent (`window.parent`, never `window.top`), which makes it fixable two ways: (a) **double-iframe** — interpose a throwaway same-origin wrapper frame so xprof's writes land there, not on the shell (proven on the POC: `/wrap/{run_key}` hosts the real xprof iframe; writes still succeed so xprof's own state read-back works); (b) **cross-origin iframe** — serve xprof from a different origin (the Iris proxy's subdomain-style routing, `endpoint_proxy.py`) so the browser blocks `window.parent` access with a caught SecurityError. (a) is the lighter POC-grade fix; (b) is stronger (also stops xprof *reading* the shell's `location`) and is the recommendation for the real service. A `sandbox` attribute does **not** help: dropping `allow-same-origin` breaks xprof's own same-origin data fetches, and keeping it still permits the parent reach.
- **The controller proxy has a 30s request timeout** ([`endpoint_proxy.py:71`](https://github.com/marin-community/marin/blob/05a1d4d474b9dccc8189c031c24cc6452c2e589a/lib/iris/src/iris/cluster/controller/endpoint_proxy.py#L71)). Mirroring a multi-hundred-MB run, or the first xprof download, far exceeds 30s — so the two slowest actions (mirror, first profile open) **must be async**: kick off work, return immediately, let the SPA poll for readiness. A synchronous `POST /api/mirror` would 504.
- **"All the data" is heterogeneous and large.** Full-fidelity metric history (parquet exports, not the sampled `history()`), config, summary, `wandb-metadata.json`, uploaded code, and the profile artifact (~275 MB). On-demand single-run fetch keeps it bounded. Full-history exports arrive as **many parquet files with drifting schemas** that must be normalized to one frame.
- **Live runs.** A still-running run has incomplete history and may not have a parquet export yet; we re-fetch on view.

## Costs / Risks

- New service to operate, plus a Vue build in the `lib/buoy/` tree and a wandb dependency in a long-running process.
- Wandb egress to GCS pulls from the open internet — bounded here because mirroring is on-demand and per-run, but a large profile or artifact set is still a real transfer.
- Running an `xprof`/TensorBoard subprocess per profile adds memory/process management to the service (lifecycle, eviction, crashes).
- A long-lived process holds a `WANDB_API_KEY`; it must be an injected secret, never committed.

## Design

A new **peer package `lib/buoy/`** (alongside `lib/finelog/`), self-contained: it depends only on `iris` (endpoint registration), `rigging` (`marin_temp_bucket`), `wandb`, `starlette`, and the `xprof` binary — **not on `marin`**. It talks to the wandb public API directly and runs the real xprof binary, so it needs no `marin.profiling` import (and introduces no reverse dependency). It boots a **Starlette** app on a single port and registers as an Iris endpoint, reachable through the controller proxy — the **finelog model**: a k8s `ClusterIP` Service added to the cluster-config `endpoints` map, served at `/proxy/system.buoy/`. (A lighter `quick_serve`-style job that self-registers via `ctx.registry.register(...)` is the alternative — see Open Questions.)

**Two concerns, cleanly split:**

1. **Mirror (compute + I/O).** `mirror_run(entity, project, run_id) -> RunMirror` pulls everything via the wandb public API and writes it under `marin_temp_bucket(30, "buoy")/{entity}/{project}/{run_id}/` (`history.parquet`, `config.json`, `summary.json`, `metadata.json`, `files/`, `artifacts/<name>/…`, and **`manifest.json` written last as the commit marker**). History: try `run.download_history_exports(dir, require_complete_history=False)`, but the spike showed a recent *finished* run returns **empty** (`contains_live_data=True`) — wandb exports to parquet lazily, so in practice `run.scan_history()` is the **primary** path, not a fallback. Both feed one `normalize_history` → `history.parquet`; the sampled `history()` is never used. (Heterogeneity is real: adjacent rows had 1302 vs 348 keys, so the union-and-coerce normalization earns its place.) **Profiles are logged artifacts**, not loose files: the mirror pulls `run.logged_artifacts()` of type `jax_profile` and records each as an xprof logdir (artifact root, which already contains `plugins/profile/<ts>/<host>.xplane.pb`). **Correctness:** a per-run in-process lock coalesces concurrent mirrors; the service runs single-replica (`replicas: 1`, `Recreate`); a run is "cached" only if `manifest.json` exists (a missing manifest or absent referenced file ⇒ re-fetch). **Re-fetch on view:** when a cached run's state was `running`, opening it re-triggers the mirror.

2. **Serve.** The Starlette app serves the Vue SPA + a JSON API. Because the proxy caps requests at 30s, the two slow actions are **async**: `POST /api/mirror` starts a background mirror and returns `202`; the SPA polls `GET …/mirror_status` until `done`, then reads from the cache. Read routes (`manifest`, `metrics`, `config`, …) serve the cached GCS copy; `GET …/manifest` re-triggers a background refresh when the cached state is `running`. `GET /api/runs` lists runs for the picker (paginated). `GET /xprof/{run_key}/{sub_path}` reverse-proxies to the run's xprof subprocess.

**xprof embedding.** On first profile view the service downloads the `jax_profile` artifact from the GCS cache to ephemeral disk (~288 MB for the canonical run; xprof reads the **local** copy, never `gs://`) and launches `xprof --logdir <local-artifact-root> --port <internal>`. Starlette reverse-proxies `/xprof/{run_key}/<sub>` → xprof `/<sub>` with **no body rewriting**, and the SPA iframes xprof under a path that ends `…/xprof/{run_key}/data/plugin/profile/`; xprof's base-href logic then resolves all assets under the proxy prefix (verified end-to-end). The iframe must be **isolated from the shell frame** so xprof's `window.parent` history/location writes don't clobber the shell's run-selection query (see Challenges) — the real service serves xprof from a cross-origin subdomain endpoint; the POC uses an intermediate same-origin wrapper frame. Subprocesses are evicted by **last-proxied-request** time (an in-flight or recently-touched process is never killed); at the `max_xprof_procs` cap with all processes active, new opens 503 rather than killing a live session. The first download is itself async (poll-for-ready) to stay under 30s. Metric plots (Plotly) and the xprof iframe are two tabs of the run view.

**Secrets / config.** `WANDB_API_KEY` and the default entity are injected as Iris env/secrets, resolved once at startup; no key lands in the repo or the cache. Note: the proxy strips `Authorization` client→upstream, so buoy cannot see a per-user identity — any future authz needs buoy's own session layer (see Open Questions).

## Testing

- **xprof-behind-proxy spike — DONE (`research.md` §Spike).** Validated against the canonical run: standalone `xprof` v2.22.3, a dumb forwarding reverse proxy under `/proxy/system.buoy/xprof/{run_key}/`, every computed asset/data URL returns 200. The remaining browser-level check (does the live iframe render trace viewer + op profile, given it pulls `ui.perfetto.dev`/`gstatic`?) is the manual integration step below.
- **Unit, mirror layer:** point `mirror_run` at a fake `wandb.Api()` (canned history/config/files/artifacts) and assert the GCS layout + `manifest.json`, the manifest-written-last commit (kill mid-write ⇒ read sees "not cached"), the concurrent-mirror coalescing, and the still-running → re-fetch path. Include a **two-export divergent-schema** history fixture to pin the normalization (column union, dtype coercion, dropped non-scalar columns). Use a `tmp_path` fsspec target, not real GCS.
- **Unit, serve layer:** Starlette `TestClient` over a pre-populated cache — assert metric-series JSON matches the parquet, the `202`+`mirror_status` polling contract, and that `/xprof/...` proxies to a stubbed backend.
- **Integration (manual) — DONE on prod.** The POC ran as a self-registering job on the **marin prod** cluster (`/serve/buoy`, reached at `/proxy/serve.buoy/`) and the canonical `marin_moe` run was driven end-to-end through the real controller proxy: metrics plotted, real xprof profile (overview + trace viewer) rendered in the iframe. The iframe-behind-proxy is the piece that can't be faked, and it surfaced two things only a live browser shows: (1) xprof's `window.parent` history clobber (fixed with the wrapper frame; see Challenges), and (2) a benign init-time `EmptyError` from xprof's own Angular bundle (`firstValueFrom` over an empty observable during first paint) — cosmetic, every data endpoint still returns 200. Two POC-only deploy gotchas worth recording: the task container mounts `/tmp` **noexec** (install the `xprof` venv under the workdir, not `/tmp`), and the synchronous mirror can exceed the proxy's 30s cap on a cold run (the mirror finishes in-container, so a reload renders) — the real service's async mirror removes this.

## Open Questions

- **Registration model:** persistent system endpoint (finelog model — k8s Service + cluster-config `endpoints` entry, `/proxy/system.buoy/`) vs. a `quick_serve`-style job that self-registers at runtime (`/proxy/{namespace}.buoy/`, no cluster-config edit but tied to a job's lifetime). Proposal picks the system endpoint for an always-on shared viewer; is that the right call, or do we want buoy launchable ad hoc?
- **xprof process model:** binary is settled (standalone `xprof`, no `--path_prefix` needed — spiked). Open: per-run subprocess (simple, more memory) vs. one server multiplexing logdirs via subdir runs. Proposal: per-run + last-request eviction; revisit if memory bites. Also: xprof needs the viewer's browser to reach `ui.perfetto.dev`/`gstatic` — acceptable for an internal tool?
- **Multiple profiles per run:** a run can log several `jax_profile` artifacts (e.g. at different steps). Proposal mirrors all and the xprof route takes a `?profile=` selector defaulting to the latest version — vs. just "latest, others ignored." Which?
- **Metrics duplicate wandb's own UI.** The user asked for in-app plots, but they overlap wandb. Worth it for single-pane-of-glass + offline-from-cache, or ship xprof-only v1 and add plots later?
- **Auth:** is "behind the Iris controller proxy" sufficient? The proxy strips `Authorization`, so per-user authz would need buoy's own session layer.
- **Region:** mirror to the service's regional bucket vs. nearest the viewer (single-region is simpler but can cost cross-region reads).
- **Artifact scope / size:** the spike's run logged a **329 MB `code` artifact** on top of the 288 MB profile — "mirror all logged artifacts" is ~600 MB/run. Proposal: mirror the `jax_profile` + small artifacts (config, requirements) by default; treat large `code`/`used_artifacts()` as opt-in. Right cut?
