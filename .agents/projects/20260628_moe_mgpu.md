# Spec: Hopper Pallas Mosaic MGPU MoE Forward/Backward Kernels

## High-level goal

Implement a Hopper GPU expert-parallel MoE MLP backend using Pallas Mosaic GPU.

The final forward implementation should expose one JAX-level MoE MLP API boundary, but internally use two main fused Pallas MGPU kernels:

1. `permute_up`: dispatch/permute routed token assignments over NVLink, then compute the expert-local `W_gate/W_up + SiLU` projection.
2. `down_unpermute`: compute the expert-local `W_down`, return/unpermute expert outputs to source token owners, and combine top-k routes deterministically.

This backend targets single-node / single-NVLink-domain expert parallelism only. Do not support NIC, InfiniBand, NVSHMEM, NCCL EP, multi-host EP, or EP > 8 in this implementation.

The algorithm should be deterministic for fixed EP, fixed hardware, fixed shapes, fixed routing, and fixed kernel configuration. In particular:

* Use deterministic assignment ordering.
* Avoid atomic accumulation for combine.
* Use unique remote write locations.
* Combine top-k routes in a fixed route-slot order.
* Treat cross-hardware bitwise equality as out of scope; the determinism target is fixed H100/NVLink execution.

## Hardware target

Use 8xH100 on `cw-us-east-02a` via Iris.

The kernel should only support EP up to 8 GPUs. The intended deployment is one expert-parallel group inside one NVLink-connected 8-GPU island.

Initial target:

```text
GPU: H100
EP: 8
top_k: 4
dtype: bfloat16 activations and weights
accumulation: WGMMA accumulator precision as provided by Mosaic/Hopper
```

## Model shape target

Per rank:

```text
x:                [T, D]
selected_experts: [T, K]
combine_weights:  [T, K]
w_gate_up:         [E_local, D, 2I]
w_down:            [E_local, I, D]
```

Concrete target values:

```text
T per device: 8 * 4096 = 32768
Total tokens across EP=8: 8 * 8 * 4096 = 262144
K: 4
D: 2560
I: 1280
E_total: 256
E_local: 32
```

Routed assignments per rank:

```text
assignments_per_source_rank = T * K = 131072
expected assignments received per destination rank ≈ T * K
```

Use capacity padding:

```text
capacity_per_rank = ceil(T * K * capacity_factor)
initial capacity_factor = 1.2
```

Overflow handling may be crude initially: fail/assert outside the kernel or drop via the existing MoE dropped-assignment mechanism. Do not implement in-kernel dynamic overflow handling in v0.

## Existing status

There is a mostly working local fused `W_gate/W_up + SiLU` Pallas Mosaic GPU prototype.

The current prototype structure is good:

* It adapts the ragged grouped GEMM structure.
* It uses two WGMMA accumulators: one for gate, one for up.
* It computes `activation(gate) * up` in the epilogue.
* It stores `[M, I]` rather than materializing `[M, 2I]`.

Known concerns / cleanup items:

1. Numerics need to be revalidated for bfloat16 using the intended fused precision semantics.
2. The current comparison against unfused `ragged_dot -> split -> activation` may be too strict because the fused kernel may avoid intermediate bfloat16 rounding of gate/up.
3. Groups with zero assigned tokens need explicit tests. The inherited `GroupInfo` logic computes `final = end - 1`, which is suspicious for empty groups even if it happens to work.
4. The current prototype’s top-level token sorting/indexing path needs cleanup before being used as the real MoE path:

   * flattened assignment IDs are not the same as token IDs;
   * `x.flatten()[sorted_indices]` is not the right way to gather token rows;
   * the metadata should distinguish assignment id, token id, route slot, expert id, destination rank, and local position.
5. Current no-comm kernel benchmarking should count both gate and up matmuls. The fused W13/SwiGLU flop count is `4 * M * D * I`, not `2 * M * D * I`.

## Overall algorithm

### Assignment model

Each source rank owns local tokens:

```text
x_source: [T, D]
```

Each token has `K` selected experts:

```text
selected_experts[t, k]
combine_weights[t, k]
```

Flatten the routing assignments:

```text
assignment_id = t * K + k
token_id = assignment_id // K
route_slot = assignment_id % K
expert_id = selected_experts[token_id, route_slot]
weight = combine_weights[token_id, route_slot]
```

For each assignment:

```text
dst_rank = expert_id // E_local
local_expert = expert_id % E_local
```

For deterministic ordering, flatten assignments in token-major, route-slot-major order and use a stable sort by global expert_id. This produces deterministic order within each expert bucket because ties retain assignment_id order.

Sorting/assignment can be done outside of pallas.

### Count exchange

Each source rank computes:

```text
send_counts[dst_rank, local_expert] = number of local assignments sent to that destination/local expert
```

Then all-gather counts across the EP axis:

```text
counts_by_src_dst_expert: [EP, EP, E_local]
```

For a destination rank `d`, compute local expert group sizes:

```text
rows_per_local_expert[e] =
    sum_src counts_by_src_dst_expert[src, d, e]
```

Compute expert-major receive offsets:

```text
expert_base[e] =
    prefix_sum(rows_per_local_expert)[e]

src_base[src, e] =
    prefix_sum over src of counts_by_src_dst_expert[src, d, e]
```

For an assignment on source rank `s` to destination rank `d` and local expert `e`, define:

```text
local_pos = index of the assignment within source-rank bucket (d, e)
remote_row = expert_base[e] + src_base[s, e] + local_pos
```

This makes every remote write unique and deterministic.

Count exchange may be done outside of pallas. Keep this simple.

### Metadata required for return/combine

The destination/expert-owner rank only needs enough metadata to send each expert output row back to a unique slot on the source rank.

Maintain minimal metadata arrays aligned with the destination expert-major row layout:

```text
recv_src_rank[row]       : int32
recv_src_assignment[row] : int32
```

where:

```text
recv_src_assignment = source_token_id * K + route_slot
```

After `W_down`, the expert-owner rank returns each row as:

```text
src = recv_src_rank[row]
a   = recv_src_assignment[row]

remote source_return_slots[a, :] = y_dispatch[row, :]
```

On the source rank, interpret:

```text
token_id   = a // K
route_slot = a % K
```

and combine in fixed route-slot order:

```text
out[token_id, :] =
    sum_k combine_weights[token_id, k] * return_slots[token_id * K + k, :]
```

This means the expert-owner rank does not need to know the route weight, token id, or route slot separately. It only needs the source rank and the source-local assignment id.

Optional debug/convenience metadata:

```text
recv_token_id[row]       : int32  # derivable from recv_src_assignment // K
recv_route_slot[row]     : int32  # derivable from recv_src_assignment % K
recv_expert[row]         : int32  # reconstructible from expert_base / rows_per_expert
recv_weight[row]         : not needed for v0; source already has combine_weights
```

For v0, keep `recv_src_rank` explicitly even though it may be reconstructible from source subranges in the expert-major layout. This makes the return path simpler and less fragile. A later optimization can derive `src_rank` from `counts_by_src_dst_expert` and row ranges if metadata bandwidth or storage matters.


### Forward dataflow

Final forward dataflow:

```text
source rank: (done in "plain jax")
    route + deterministic assignment metadata
    count exchange
    compute offsets

kernel 1: permute_up
    remote-write x[token_id, :] into destination recv_x[remote_row, :]
    remote-write row metadata into destination recv_meta[remote_row]
    wait for all inbound writes
    run local grouped W_gate/W_up + SiLU over recv_x by local expert
    output hidden: [capacity_per_rank, I]

kernel 2: down_unpermute
    run local grouped W_down over hidden by local expert
    remote-write each result row to source return slot [token_id, route_slot, :]
    wait for all return writes
    combine return slots in fixed k order:
        out[token] = sum_k combine_weights[token, k] * return_slots[token, k]
```

Do not use remote atomic add for combine in the initial implementation.

## Kernel 0 metadata outputs revision

Change the metadata outputs to:

```text
assignment_ids_sorted: [T*K]          # source-local assignment id = token_id * K + route_slot
token_ids_sorted:      [T*K]          # assignment_ids_sorted // K
expert_ids_sorted:     [T*K]
dst_ranks_sorted:      [T*K]
local_experts_sorted:  [T*K]
local_pos_sorted:      [T*K]          # position within source bucket (dst_rank, local_expert)
send_counts:           [EP, E_local]
global_counts:         [EP, EP, E_local]
```

`route_slots_sorted` is optional debug metadata and is derivable from:

```text
route_slots_sorted = assignment_ids_sorted % K
```

`combine_weights_sorted` is not needed by the dispatch/up kernel. Keep `combine_weights` on the source rank for the final local weighted combine.

For determinism, flatten assignments in token-major, route-slot-major order:

```text
assignment_id = token_id * K + route_slot
```

Then use a stable sort by global expert id:

```text
assignment_ids_sorted = stable_argsort(expert_ids_flat)
```

This produces deterministic ordering by:

```text
global_expert_id, then assignment_id
```

which is equivalent to:

```text
dst_rank, local_expert, token_id, route_slot
```

provided `expert_id = dst_rank * E_local + local_expert`.


Important correction:

```text
assignment_ids_sorted = argsort/lexsort over flattened assignments
token_ids_sorted = assignment_ids_sorted // K
route_slots_sorted = assignment_ids_sorted % K
tokens_sorted = x[token_ids_sorted]
```

Do not use:

```text
x.flatten()[assignment_ids_sorted]
```

That is wrong because assignment IDs index `[T*K]`, not flattened token elements.

For determinism, use a stable ordering.

Dropped assignments:

* Do not worry about this for v0. Leave a todo.
* If dropped routes are encoded as `expert_id = -1`, mask them before `bincount`, sort, and kernel dispatch. 
* Either remove dropped assignments from the sorted route list or route them to a dummy location with zero weight and ensure they do not affect output.
* Match existing Grug MoE dropped-assignment semantics.

## Reference implementation

Before MGPU kernels, create an explicit reference for each stage using existing JAX operations.

Reference forward:

```text
1. Flatten and sort routed assignments by destination/local expert.
2. Use all-gather or all-to-all/ppermute-style movement to materialize destination expert-major x.
3. Run reference ragged W13.
4. Split gate/up.
5. Compute SiLU(gate) * up.
6. Run reference ragged W2.
7. Return/unpermute expert outputs to [T, K, D] slots.
8. Combine in fixed route-slot order.
```

This reference should be intentionally boring and easy to inspect, not fast.

Use the existing `ep_ring.py` path to build the first reference, then add a smaller/direct reference if needed for testing.

## Kernel 1b: local fused W13/SwiGLU kernel

This is the current no-communication fused kernel, cleaned up and made production-quality.

Signature sketch:

```python
def ragged_w13_swiglu_mgpu(
    recv_x: Array,           # [M_or_capacity, D]
    w_gate_up: Array,        # [E_local, D, 2I]
    rows_per_expert: Array,  # [E_local]
    *,
    activation_fn,
    config: MoeMgpuConfig,
) -> Array:                  # [M_or_capacity, I]
    ...
```

Algorithm:

```text
for each expert-major M tile and I/N tile:
    load lhs tile [block_m, block_k]
    load gate weights [block_k, block_n]
    load up weights [block_k, block_n]
    WGMMA into gate_acc
    WGMMA into up_acc
    hidden = SiLU(gate_acc) * up_acc
    store hidden tile
```

Implementation details:

* Output shape is `[M, I]`, not `[M, 2I]`.
* Use two accumulators.
* Assume W13 layout is non-interleaved:

  * `w_gate_up[..., :I]` is gate;
  * `w_gate_up[..., I:]` is up.
* Keep the current power-of-two/log2 static-store ladder. You may extract it a helper if useful.
* Add `actual_size > 0` guards if empty groups trigger invalid loads/stores or wasted work.

Numerics:

* For bfloat16, compare against a reference with matching fused semantics.
* The unfused reference may round gate/up before activation; the fused kernel may apply activation in accumulator precision and cast once.
* Add tests for both:

  1. fused-semantics reference;
  2. unfused-semantics reference with relaxed tolerance.

Suggested initial tolerances for bfloat16:

```text
linear epilogue sanity test: tighter, e.g. rtol=2e-3, atol=1e-2
fused SiLU vs matching fused reference: rtol=5e-3 to 1e-2, atol=0.1
fused SiLU vs unfused reference: allow relaxed rtol up to 1e-2, inspect max relative error
```

Check for "clumpiness" in the error matrix. If errors are consistently much larger in one block that is a sign for concern.

Do not accept NaNs/Infs.

## Kernel 1c: fused `permute_up` MGPU kernel

This is the first final forward kernel.

Signature sketch:

```python
def permute_up_mgpu(
    x: Array,                         # [T, D]
    assignment_ids_sorted: Array,      # [T*K], source-local assignment id = token_id * K + route_slot
    token_ids_sorted: Array,           # [T*K], assignment_ids_sorted // K
    dst_ranks_sorted: Array,           # [T*K]
    local_experts_sorted: Array,       # [T*K]
    local_pos_sorted: Array,           # [T*K], position within source bucket (dst_rank, local_expert)
    global_counts: Array,              # [EP, EP, E_local]
    w_gate_up: Array,                  # [E_local, D, 2I]
    *,
    expert_axis: str,
    config: MoeMgpuConfig,
) -> tuple[Array, RecvMetadata]:
    ...
```

Outputs:

```text
hidden:              [capacity_per_rank, I]
recv_src_rank:       [capacity_per_rank]
recv_src_assignment: [capacity_per_rank]
rows_per_expert:     [E_local]
```

where:

```text
recv_src_assignment = source_token_id * K + route_slot
```

Initial algorithm:

```text
Phase 1: dispatch
    for each source-local sorted assignment:
        token_id = token_ids_sorted[a]
        src_assignment = assignment_ids_sorted[a]
        dst = dst_ranks_sorted[a]
        local_expert = local_experts_sorted[a]
        local_pos = local_pos_sorted[a]

        remote_row =
            expert_base[local_expert]
          + src_base[src_rank, local_expert]
          + local_pos

        remote-write x[token_id, :] into destination recv_x[remote_row, :]

        remote-write minimal return metadata:
            recv_src_rank[remote_row] = my_rank
            recv_src_assignment[remote_row] = src_assignment

    signal destination after dispatch is complete

Phase 2: wait
    wait until all source ranks have signaled expected arrivals

Phase 3: local compute
    run local ragged W_gate/W_up + SiLU over recv_x using rows_per_expert
    output hidden
```

`combine_weights` are not needed by `permute_up_mgpu`. They remain on the source rank and are used later by `down_unpermute_mgpu` when combining returned route slots in fixed route-slot order.


This first fused version does not need to overlap communication and compute. It may be one Pallas kernel with two phases or a wrapper around the permute-only and local-up kernels. The final target should be one kernel, but correctness matters first.

Future overlap:

* Signal per source/expert/block instead of once per peer.
* Allow compute workgroups to consume ready expert blocks as soon as they arrive.
* Use a static ready schedule derived from counts, not a fine-grained dynamic global queue.

Correctness:

* Compare `hidden` and metadata against:

  * reference permute + reference ragged W13/SwiGLU;
  * permute-only MGPU + local W13/SwiGLU.
* Include zero-token expert tests.
* Include all assignments going to one remote rank.
* Include all assignments staying local.
* Include random routing at target shape.

Performance target:

* Build roofline using:

  * `F_up = 4 * M_recv * D * I` FLOPs per rank;
  * dispatch bytes approximately `M_sent * D * sizeof(dtype)` plus metadata.
* First target: 50% of the relevant roofline for the compute-heavy portion.
* Do not over-optimize until correctness is stable.

## Kernel 2b: `unpermute_mgpu` revision

Signature sketch:

```python
def unpermute_mgpu(
    y_dispatch: Array,           # [capacity_per_rank, D]
    recv_src_rank: Array,        # [capacity_per_rank]
    recv_src_assignment: Array,  # [capacity_per_rank]
    rows_per_expert: Array,      # [E_local]
    combine_weights: Array,      # [T, K], source-local
    *,
    expert_axis: str,
    config: MoeMgpuConfig,
) -> Array:                      # [T, D]
    ...
```

Initial algorithm:

```text
expert-owner rank:
    for each valid expert-major row:
        src = recv_src_rank[row]
        a = recv_src_assignment[row]

        remote-write y_dispatch[row, :] into source return_slots[a, :]

    signal source rank when done

source rank:
    wait for all returns

    for each token_id:
        out[token_id, :] = 0
        for route_slot in 0 .. K-1:
            a = token_id * K + route_slot
            out[token_id, :] += combine_weights[token_id, route_slot] * return_slots[a, :]
```

For v0, materialize:

```text
return_slots: [T*K, D]
```

or equivalently:

```text
return_slots: [T, K, D]
```

Prefer `[T*K, D]` internally because `recv_src_assignment` indexes it directly.

Do not use remote atomic add.

This is memory-heavy but simple and deterministic.

Potential later optimization:

* Locally pre-combine same-source/same-token rows only if it preserves deterministic semantics.
* Avoid duplicate sends for top-k experts on the same destination rank. This is a future optimization, not v0.

Correctness:

* Compare against reference unpermute/combine.
* Include top-k routes returning from multiple ranks.
* Include repeated destination rank among top-k routes.
* Include dropped/zero-weight routes.

Sufficiently large token counts with uniform routing should address all of these but include focused tests.

## Kernel 2c: fused down_unpermute MGPU kernel

Signature sketch:

```python
def down_unpermute_mgpu(
    hidden: Array,           # [capacity_per_rank, I]
    recv_src_rank: Array,    # [capacity_per_rank]
    recv_token_id: Array,    # [capacity_per_rank]
    recv_route_slot: Array,  # [capacity_per_rank]
    recv_weight: Array,      # [capacity_per_rank]
    rows_per_expert: Array,  # [E_local]
    w_down: Array,           # [E_local, I, D]
    combine_weights: Array,  # [T, K]
    *,
    expert_axis: str,
    config: MoeMgpuConfig,
) -> Array:                  # [T, D]
    ...
```

Initial fused algorithm:

```text
Phase 1: local W2
    compute expert-major W2 output tiles

Phase 2: return/unpermute
    write W2 outputs into remote source return_slots[token, route_slot, :]
    signal source ranks

Phase 3: combine
    wait for all returns
    source rank performs fixed-order top-k weighted sum
```

## Backward pass

The final backend should use a single custom VJP at the public MoE API boundary.

Do not expose Pallas scratch tensors in the public API.

### Backward of down_unpermute

Forward:

```text
hidden -> W2 -> return slots -> weighted combine -> out
```

Backward inputs:

```text
dout: [T, D]
```

Conceptual backward:

```text
d_return_slots[token, k, :] = dout[token, :] * combine_weights[token, k]
d_combine_weights[token, k] = dot(dout[token, :], return_slots[token, k, :])

permute/gather d_return_slots back to expert-owner rows:
    d_y_dispatch[row, :] = d_return_slots[token_id, route_slot, :]

W2 backward:
    d_hidden_e += d_y_e @ W2_e.T
    dW2_e += hidden_e.T @ d_y_e
```

Milestone 12 target:

* Create backward kernel for `down_unpermute`.
* Check correctness against JAX autodiff/reference.
* Aim for 40% of roofline initially.

### Backward of permute_up

Forward:

```text
x -> permute -> recv_x
recv_x @ W_gate/W_up -> gate/up
hidden = SiLU(gate) * up
```

Backward inputs:

```text
d_hidden: [capacity_per_rank, I]
```

Need:

```text
d_recv_x
d_w_gate_up
```

Then reverse-permute:

```text
d_x[source token] += d_recv_x[row]
```

Because top-k assignments duplicate source tokens, `d_x` is a sum over route slots. To keep determinism, avoid remote atomics initially:

* return gradients into deterministic `[T, K, D]` slots;
* sum route slots locally in fixed order.

For SiLU derivative:

```text
hidden = silu(gate) * up
d_up = d_hidden * silu(gate)
d_gate = d_hidden * up * silu'(gate)
```

Need gate/up values. Initial options:

1. Save gate/up from forward. Simple but memory-heavy.
2. Recompute gate/up in backward from `recv_x` and `w_gate_up`. Preferred after correctness.
3. Save only `recv_x` and recompute gate/up. Reasonable first optimized path.
4. Re-dispatch `x` in backward to recreate `recv_x`. More compute/comm but less residual memory.

Initial backward may choose the simplest correct option. Longer-term preference: recompute gate/up rather than store `[M, 2I]`.

Milestone 11 target:

* Create backward kernel for `permute_up`.
* Check correctness against JAX autodiff/reference.
* Aim for 40% of roofline initially.

FLOP formulas per rank:

```text
M_recv = number of valid routed rows received by this rank

W13/SwiGLU:
    F_up = 4 * M_recv * D * I

W2:
    F_down = 2 * M_recv * I * D

Full expert FFN compute:
    F_total = 6 * M_recv * D * I
```

Approximate communication bytes per source rank:

```text
dispatch bytes ≈ (T * K) * D * sizeof(dtype)
return bytes   ≈ (T * K) * D * sizeof(dtype)
metadata bytes ≪ activation bytes
```

Do not optimize metadata exchange early; it is tiny relative to activation movement.
