## Generator code path

The qsplit weights were generated by the nextgen static design path, not by the
plain sampler:

1. `experiments/domain_phase_mix/nextgen_experiment.py` builds a `LoopConfig`
   with `design_policy="static_d_optimal"`.
2. `experiments/domain_phase_mix/nextgen/design.py::plan_new_runs` dispatches to
   `_sample_with_static_selector`.
3. `_sample_with_static_selector` calls
   `experiments/domain_phase_mix/static_batch_selection.py::prospective_d_optimal_selection`.
4. `prospective_d_optimal_selection` builds a Sobol candidate pool with
   `sobol_weight_configs`, fits a DSRE-CEQ anchor on the current state, evaluates
   the candidate Jacobian, and greedily selects rows by D-optimal log-det gain.

The candidate pool construction is:

- dimension: `n_phases * n_domains = 2 * 39 = 78`
- Sobol generator: `scipy.stats.qmc.Sobol(..., scramble=True, seed=42)`
- simplex transform: `phase_raw = -log(sobol_uniforms)`, then normalize within
  each phase
- post-processing: the experiment sampler's `_normalize`, including the
  `min_weight=2e-4` threshold
- distance rejection: average phase TV distance at least `0.001` from existing
  and already accepted configs

The D-optimal selection is greedy log-det over the fitted anchor Jacobian with
the default `DEFAULT_INFO_RIDGE = 1e-3`.

## Plain sampler was not the source

The committed Mar 17/18 plain sampler configuration for this experiment was:

```python
DirichletSamplingParams(
    strategy=SamplingStrategy.DIRICHLET,
    temp=0.5,
    min_strength=0.1,
    max_strength=0.5,
    min_weight=2e-4,
    min_config_distance=0.001,
)
```

That path also enforces `max_ratio=15` relative to natural proportions. The
persisted qsplit weights violate those bounds. For example, in `run_00002`
phase 0:

```text
dolma3_wikipedia planned weight = 0.17946479387685207
Mar 17/18 Dirichlet upper bound = 0.007877708819127058
```

Replaying the plain Dirichlet sampler at commits `14ead23c1` and `c43345016`
with seed 42 gives a completely different sequence, with max absolute mismatch
about `0.654` against the persisted `planned_runs.json`.

## Reproduction check

Replaying the Mar 17 static D-optimal code path with:

- the two baseline rows from GCS state,
- `lm_eval/mmlu_5shot/bpb` as the objective,
- seed 42,
- Sobol pool size 2048,
- target budget `6325183647689`,
- phase fractions `[0.8, 0.2]`,

recovers the same selected Sobol candidate set for 233 of the 238 persisted
planned runs. The first selected candidate matches exactly, including the large
phase-0 Wikipedia weight.

The remaining 5-row discrepancy is likely because the exact launch-time
prestate metrics were mutable and are no longer preserved separately. The
current GCS `state/loop_state.json` has later additions and may have refreshed
baseline metrics. The persisted `planned_runs.json` remains the authoritative
weight table.

## Conclusion

The qsplit240 weights were generated by Sobol candidate sampling followed by
greedy D-optimal selection using the static nextgen design path. D-optimal was
definitely involved. The plain Dirichlet sampler was only the source of
experiment constraints and normalization helpers, not the direct qsplit point
generator.
