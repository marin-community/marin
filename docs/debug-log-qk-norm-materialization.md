# Debugging log for Delphi QK-norm prefix materialization

Audit whether the Delphi prefix materialization path is fixed after the
previous 70% materialized checkpoints dropped Qwen3 `q_norm` / `k_norm`
arrays.

## Initial status

User reported that the previous materialized 70% checkpoints were generated by
buggy code that dropped QK-norm arrays. The branch is clean and up to date at
`eb6b3cc01d`. The most recent logbook entry says the fix is to rematerialize
70% and 80% prefixes into new `-qwen3` roots, then flip YAML only after schema
verification.

## Hypothesis 1: old untyped Delphi executor info still decodes as Llama

The original failure was caused by decoding old untyped Delphi `.executor_info`
model payloads as Llama, which built a model tree without Qwen3 QK-norm
leaves. The audit must prove this path now decodes as Qwen3 or fails before
training.

## Results

Confirmed and strengthened.

- `scripts/materialize_delphi_prefix_checkpoint.py` now forces old untyped
  Delphi model payloads through `type: qwen3` and refuses to continue unless
  draccus decodes a `Qwen3Config`.
- Added a source checkpoint schema check to `validate_checkpoint_io()`, so a
  source checkpoint missing Qwen3 `q_norm` / `k_norm` arrays fails before any
  executor step is submitted.
- Added `MaterializationTrainConfig` and `run_materialization_train()`, wrapping
  `run_levanter_train_lm()` so every intended destination checkpoint is checked
  after training. If any saved step is missing QK-norm arrays, the executor step
  fails instead of leaving a bad-looking-successful materialization.
- Added regression tests that prove:
  - old untyped `.executor_info` decodes as `Qwen3Config`;
  - the decoded Qwen3 attention config has `qk_norm` enabled even when the old
    payload contains `use_qk_norm: False`;
  - source checkpoint schema validation is called and fails closed;
  - all materialized destination checkpoints are checked after training;
  - destination schema failures propagate.

  passed: `102 passed`.
- `uv run --with pytest --with pytest-timeout python -m pytest tests/midtraining tests/test_materialize_delphi_prefix_checkpoint.py -q --timeout=180`
  passed: `155 passed`.
- Safe GCS metadata/schema check for all seven planned `*-prefixes-qwen3`
  materializations passed. Each source `.executor_info` decoded as
  `Qwen3Config`, each canonical source checkpoint passed schema, and each
  us-east5 mirror was present and schema-clean.
- Safe GCS schema check rejected the three known-bad invalidated 70% materialized
  checkpoints (`3e18`, `3e19`, `3e20`) with missing `q_norm` and `k_norm`.
- `./infra/pre-commit.py --all-files --fix` passed, including pyrefly.

## Hypothesis 2: cooldown launch path might still accept an untyped/degraded checkpoint

The true-cooldown path must independently refuse any source checkpoint or model
config that could replay this bug class.

## Results

Existing defenses are active:

- `model_config_for_base()` in the true-cooldown launcher returns explicit
  `type: qwen3` and fails unless the heuristic produced `Qwen3Config`.
- `validate_midtrain_spec()` rejects missing or non-string
  `model_config["type"]`.
- `preflight()` checks staged cooldown checkpoint OCDBT keys before launch.
- The true-cooldown launcher checks all saved permanent checkpoint dirs after a
  successful run.

The targeted and broad test runs above cover these defenses.
