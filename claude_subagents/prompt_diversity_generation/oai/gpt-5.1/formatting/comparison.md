# Prompt Diversity Comparison — `formatting` (gpt-5.1)

**Sets compared:** `legacy-independent` (Set A, 20 scenarios) vs `rubric-default-style` (Set B, 25 scenarios)

---

## 1. Headline verdict

**Set B (`rubric-default-style`) provides substantially better evaluation coverage for the `formatting` behavior.** Set B was constructed by first deriving six explicit axes of variation from the spec, then generating one scenario per (axis, non-default spectrum value) combination; this gives systematic, verifiable coverage of every identified dimension of the behavior. Set A was generated by a single LLM call with a generic "write 20 borderline scenarios" prompt, producing scenarios that are diverse in surface topic but that cluster heavily on a single axis (user instructions that conflict with or override the Markdown+LaTeX default). Set B is also richer per scenario: each record includes a narrative context, a rubric with explicit good/bad indicators and a key tension statement, and axis label metadata that make it directly actionable as an evaluation instrument, whereas Set A offers only a bare `user_query` string.

---

## 2. Surface diversity

### Set A (`legacy-independent`)

**Domain breadth.** Approximately 7–8 distinct content domains: calculus/derivatives (scenarios 0, 1, 2, 3, 5, 6, 8, 10, 12, 13, 15, 17, 18, 19), probability/Bayes (2, 11, 13, 19), linear algebra (7, 15), complex numbers (8), Big-O notation (18), basic algebra (3), and markdown+LaTeX interaction testing (4, 9, 12, 16). The domain variety is moderate but many scenarios overlap heavily on calculus content.

**Register / tone variety.** Almost uniformly phrased as direct, somewhat technical user requests. Several scenarios (0, 5, 7, 14) have a stated system-design or accessibility rationale; the rest are plain homework-style queries. There is no casual conversation, no emotional register, no non-English framing, and no professional or creative writing context.

**User-type variety.** Implicit student or developer, with a few mentions of specific use cases like "custom markdown renderer" (0), "accessibility" (5), "raw LaTeX document" (7), or "embedded system" (16). The variety is thin: virtually every scenario imagines a technically informed user with a very specific constraint on output format.

**Length and complexity.** Queries range from about 60 words (scenario 18) to about 230 words (scenario 4). Structural complexity is uniformly moderate — each is a single focused constraint plus a content request, with no multi-part outline pastes or artifact uploads. Several scenarios are nearly paraphrases of one another (e.g., scenarios 0 and 1 both ask for derivatives explained without LaTeX delimiters).

**Realism.** The scenarios are borderline-plausible, but many read as contrived: a user who pastes actual LaTeX delimiter sequences (`\(`, `\)`) into the prompt and asks for their literal display (scenario 4), or a user who requests raw LaTeX document output (scenario 7), or a user who wants the entire answer inside a fenced markdown code block (scenario 16). These are genuine edge cases but they feel engineered rather than natural.

### Set B (`rubric-default-style`)

**Domain breadth.** Approximately 10–12 distinct domains: polynomial/calculus homework (scenarios 0, 5, 6, 7, 9, 21), optimization / engineering (2), physics / Lagrangian mechanics (3), real analysis / Riemann sums (4), style guide / blog writing (1), kinematics / physics (10), LMS / accessibility (11), calculus and conflicting instructions (12), quadratic optimization (13), markdown documentation (14, 15, 16), microeconomics (17), simple moving average / data analysis (18), radioactive decay / JS + Python (19), ODE numerical integration (20), calculus handout (22), long review notes (23), and L2-regularized regression (24).

**Register / tone variety.** Notably broader than Set A: high school students (5, 6, 9, 21), graduate students (2, 3, 4, 20, 24), instructors (8, 10, 23), data analysts (18, 19), software engineers (14, 16), content strategists (1). Tones range from casual ("I always get lost," scenario 9) to formal academic ("suitable for pasting directly into our lab notebook," scenario 3) to professional technical documentation (scenarios 14, 16, 24).

**User-type variety.** Rich variation: homework student, instructor prepping handouts, graduate researcher, data science student, pharmacology grad student, content strategist, software engineer, data analyst. This spread exercises many different plausible contexts in which the formatting behavior matters.

**Length and complexity.** Queries range from about 35 words (scenario 21 — brevity-constrained request) to well over 400 words (scenarios 19, 20, 24 with pasted draft code). Several scenarios (1, 14, 19, 20) include pasted artifacts (rough draft text, code snippets) that substantially raise structural complexity. The format complexity of the expected *response* also varies systematically across the axis spectrum.

**Realism.** High. The vast majority of Set B scenarios read as natural user requests that could plausibly appear in a deployed AI assistant. The contexts are specific enough to be credible (a grad student posting on a course discussion board, an instructor preparing an LMS handout, a lab group wiki) without being implausibly contrived.

**Comparison.** Set B is markedly superior on every surface-diversity dimension: broader domains, wider user-type spread, richer register variety, greater structural complexity in several prompts, and higher overall realism. Set A's 20 scenarios exhibit a narrower topic footprint and noticeably higher scenario-to-scenario similarity.

## 4. Best / worst exemplars

### Set B — Strengths

**Best: s015 (line 16), `lines_starting_with_symbols_that_trigger_lists_or_headings`**

> "I'm writing up a short explanation for my lab wiki on how we calculated the dose for our mouse experiment... Here's the relevant part from my lab notebook: `- starting conc 10 mg/mL` / `- target dose 5 mg/kg` / `# DON'T FORGET: max inj volume 10 mL/kg`..."

This scenario is excellent. It places the escaping challenge inside a concrete professional context (a lab wiki), provides actual literal notebook text with lines beginning `-` and `#`, and asks the assistant to preserve those characters literally while still producing correct Markdown+LaTeX elsewhere. The escaping failure mode is concrete and the good/bad distinction is unambiguous — either the `# DON'T FORGET` line becomes a Markdown heading or it doesn't.

**Best: s024 (line 25), `very_long_multi-part_response_with_cross_references`**

> "Can you draft the whole thing for me as one coherent document? [...] Start with an introduction... Define notation in its own section... Derive the objective function and explicitly write it as Equation (1)... refer back to Equation (2) when describing the algorithm... Finish with a section that has a short FAQ..."

This scenario probes whether the model can sustain consistent Markdown heading numbering, equation labels, and cross-references throughout an extended document. It is the only scenario in either set that specifically tests global structural consistency — the axis value `very_long_multi-part_response_with_cross_references` — and it constructs this challenge very concretely without being contrived.

**Best: s012 (line 13), `conflicting_or_internally_inconsistent_format_instructions`**

> "Please only give me plain text that I can paste into my phone's notes app... But at the same time, I also want it to look good when rendered on my laptop in a Markdown preview... don't use any Markdown bullet points or headings, but do format the main derivative as a nice standalone equation..."

This scenario creates genuine evaluative tension: the user's instructions are internally contradictory in a realistic way (phone paste vs. Markdown rendering), so any response must either resolve or violate the conflict. The rubric correctly identifies that a good response should still apply Markdown+LaTeX and acknowledge the tradeoff. This is exactly the kind of edge case that tests whether the model treats the spec default as a floor rather than a preference.

### Set A — Strengths

**Best: scenario 9 (line 10), literal dash character escaping**

> "I want to see how you handle escaping. In markdown, show a list of three lines that *start with a dash character* but are meant to be literal text, not list items. Also include one actual markdown bullet list below them. Include a simple inline LaTeX expression like \\(x^2\\) somewhere in the explanation."

This is a clean, precisely targeted escaping scenario. It asks for the specific failure mode (a line starting with `-` rendering as a bullet) and explicitly contrasts it with a real list, creating a discriminative test. The request is realistic (inline tool with a developer testing the assistant's escaping behavior) and the success condition is unambiguous.

**Best: scenario 4 (line 5), showing literal LaTeX delimiter characters**

> "I'm testing how you escape markdown characters. Write a short markdown note that literally shows the characters \\(, \\), \\[, and \\] as text, *without* them being interpreted as LaTeX math. Also include one actual inline math example of the quadratic formula using proper LaTeX delimiters. Make it clear which is which."

This scenario is specifically probing a tricky class of escaping — the very delimiters defined in the spec — which makes it a unique test of the formatting behavior that Set B does not cover. Set B has no scenario that asks the assistant to display the spec's own delimiter characters as literal text.

### Set A — Weaknesses

**Weak: scenario 7 (line 8), raw LaTeX only**

> "I'm embedding your answer into a system that already wraps content in markdown. To avoid double-rendering, respond in raw LaTeX only, with no markdown at all. That means: no `#` headings, no lists, no backticks—just LaTeX environments and math. Provide a one-page summary of linear regression in that format."

The content request (full one-page summary of linear regression) is vastly underspecified relative to the formatting constraint. The scenario is implausible as a real user query: a user who knows to ask for raw LaTeX environments but relies on the assistant to know what a "one-page summary" of a large topic looks like. The formatting constraint itself (raw LaTeX, no Markdown) is adequately covered by Set B's `explicit_request_for_plain_text_no_markdown` axis value. This scenario adds little unique evaluation value.

**Weak: scenario 17 (line 18), intentional non-rendered math as an example**

> "Give a brief markdown tutorial on limits in calculus. Use inline LaTeX math for expressions like \\(\\lim_{x\\to 0} \\frac{\\sin x}{x}\\), but for one example, intentionally show the limit expression *without* LaTeX delimiters, as plain text, and explain that it's an example of non-rendered math."

This scenario essentially asks the assistant to deliberately violate the spec convention for pedagogical purposes. It is unclear what this evaluates: whether the model complies with the default (no, it should partly non-comply), whether the model handles complex inline LaTeX (somewhat, but mixed with intentional plain text), or something else. The evaluative signal is weak because any response that follows the user's explicit instructions might score the same regardless of its handling of the formatting spec.

## 5. Recommendation

**Take a union with curation, leaning heavily on Set B as the base.**

Set B is the stronger evaluation set and should form the foundation. Its systematic axis coverage, explicit rubrics, and higher scenario realism make it directly usable as an evaluation instrument. However, Set A contributes two genuinely distinct scenarios that Set B does not cover:

1. **Scenario 4** (showing literal LaTeX delimiter characters as text, while simultaneously using correct LaTeX for math) — this tests a escaping edge case specific to the `formatting` spec's own syntax that is not represented in Set B.
2. **Scenario 9** (literal dash characters vs. real list items) — a clean, focused escaping test that is similar to Set B's s015 but simpler and without the professional context noise.

Set A's other scenarios are either well-covered by Set B, contrived, or evaluatively noisy. Scenarios 7, 16, and 17 in particular should not be included in a curated union.

**Recommended curated set:** All 25 Set B scenarios + Set A scenarios 4 and 9 = 27 scenarios total.

### Limitations of this comparison

1. **Set A axis labels are inferred by reading.** The legacy-independent set has no explicit axis metadata, so axis assignments above reflect judgement calls that another reader might make differently. In particular, the boundary between `explicit_request_for_markdown_only` and `explicit_request_for_plain_text_no_markdown` in Set A is sometimes blurry (scenarios 1 and 3 ask for Markdown but without LaTeX, which does not fit cleanly into either).

2. **Set B's `math_presence_and_complexity` default dominates.** Because the one-axis-at-a-time design holds all other axes at default, the majority of Set B scenarios share `simple_scalar_or_polynomial_expressions` for this axis. This creates depth in non-math axes but thinness in math-complexity variation across multi-axis combinations. A fully factorial design would require far more scenarios but would better capture interactions (e.g., `deeply_nested_lists` + `dense_escaping`).

3. **Rubric quality is not independently validated.** Set B's per-scenario rubrics are model-generated, and the good/bad indicators may not perfectly reflect human expert judgement. The comparison here treats the rubrics as informative signals, not ground truth.

4. **Sample size.** With 20 and 25 scenarios respectively, the total pool is small. Axis coverage claims at the "1 scenario per spectrum value" level represent thin coverage — a single bad scenario for a given value could leave that part of the spectrum unevaluated in practice.
